## 개발 목적

- 현재 `Codex CLI + local file-backed runtime` 기본선을 유지한 채, 다음 확장 축의 실제 구현 우선순위를 정리한다.
- `direct upstream parity`, `pane backend`, `remote transport`를 한 번에 다 여는 것이 아니라, 어떤 순서와 어떤 최소 contract로 접근해야 하는지 고정한다.
- 현재 `agent-team` 구조에서 재사용 가능한 레이어와 새로 추상화해야 할 레이어를 분리한다.
- 향후 명시 수요가 생겼을 때 바로 들어갈 수 있는 구현형 dev-plan 기준선을 만든다.

## 문서 기반 제약 사항

- 현재 제품 기본선은 `local runtime + codex-cli + file-backed store + TUI/CLI operator`다.
- `upstream parity`는 현재 surface를 버리는 작업이 아니라, 이미 존재하는 upstream/alt runtime path와의 동작 차이를 줄이는 작업이어야 한다.
- `pane backend`는 UI/운영 surface가 아니라 launch/backend abstraction 계층 문제로 다뤄야 한다.
- `remote transport`는 transport contract, lifecycle, logs, heartbeat, recovery를 모두 포함하는 별도 계층 문제로 봐야 한다.
- 이번 workstream의 성공 기준은 “무언가를 많이 구현함”이 아니라, **다음 실제 구현 workstream에서 안전하게 착수할 수 있는 최소 설계/우선순위/검증 기준을 잠그는 것**이다.

## 현재 출발점 요약

- single-team / multi-team / richer global ops surface / burn-in / release checklist 기본선은 이미 갖춰졌다.
- runtime recovery, planner-first bootstrap, staged closeout, representative bootstrap-complete path도 known-good baseline으로 올라왔다.
- 따라서 다음 확장 축은 제품 기본선 보완이라기보다 **선택적 parity / transport 확장** 문제다.
- 특히 현재 backlog 기준으로는 `direct upstream parity / pane backend / remote transport`가 다음 후보이며, 이는 구현 전에 먼저 최소 contract와 단계적 범위를 고정하는 편이 안전하다.

### Phase 2 결과 메모

최소 abstraction contract는 다음 4개로 정리했다.

1. **launcher abstraction**
   - local / pane / remote 중 어떤 실행 바인딩을 쓰더라도 동일한 goal bootstrap 입력을 받아야 한다.

2. **runtime turn bridge abstraction**
   - Codex CLI, upstream CLI, future remote bridge가 공통으로 구현해야 할 turn contract를 유지한다.

3. **observability abstraction**
   - status / logs / transcript / files / heartbeat는 backend가 바뀌어도 같은 semantics를 유지해야 한다.

4. **recovery abstraction**
   - shutdown / reopen / resume / stale cleanup는 transport가 달라도 같은 ownership 규칙을 따른다.
