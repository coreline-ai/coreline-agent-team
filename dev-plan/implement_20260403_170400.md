## 개발 목적
이번 개발의 목적은 실제 `Codex CLI` 장시간 turn 중에 `atcli` / `attach` / `status` / `watch`에서 작업이 멈춘 것처럼 보이는 문제를 줄이는 것이다.

핵심은 실행 엔진 자체를 바꾸는 것이 아니라, **turn 실행 중 heartbeat / 상태 메타데이터 / 사용자 표시 문구를 보강**해서 사용자가 현재 시스템이 실제로 일하는 중인지, 정체 상태인지 더 명확하게 판단할 수 있게 만드는 것이다.

## 개발 범위
1. live `codex-cli` turn 실행 중 runtime state가 너무 오래 갱신되지 않는 구간을 줄인다.
2. worker가 현재 무엇을 하고 있는지 드러내는 최소 메타데이터를 저장한다.
3. `attach`, `status`, `watch`, 앱 대시보드에서 long-running turn / settling / stale 상태를 더 분명하게 보이게 한다.
4. 장시간 turn과 정체처럼 보이는 재현 케이스를 테스트/실검증 절차에 반영한다.
5. 기존 `local`, `codex-cli`, `upstream` 표면을 유지하면서 backward-compatible 하게 개선한다.

## 제외 범위
- runtime 종류 추가 또는 direct API 연동
- preset/role 체계 재설계
- leader approval UI 대규모 리디자인
- pane backend / remote transport 추가
- cosmetic 중심의 TUI 전체 개편

## 잔여 리스크 / 후속 과제
- live `codex exec` 응답 지연 자체는 제거할 수 없으므로, 이번 작업은 지연을 숨기지 않고 **관찰 가능하게 만드는 것**을 목표로 한다.
- stale 기준이 너무 짧으면 false alarm이 늘고, 너무 길면 UX 개선 효과가 약해질 수 있다.
- upstream CLI bridge에도 같은 상태 메타데이터 계약을 확장할지 후속 검토가 필요하다.
