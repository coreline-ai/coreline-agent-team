attach/status/TUI와 필요한 경우 dedicated read-only CLI log surface 기준의 읽기 전용 log tail 가시성 강화, bounded log reader 규칙 정리, selected teammate 기준 stdout/stderr 확인 UX, 관련 테스트/문서 갱신에 한정한다.

## 문서 기반 제약 사항
- 프로젝트의 표준 runtime 경로는 항상 `Codex CLI`이며, direct API 기반 모델 경로 추가는 금지한다.
- 직전 workstream(`implement_20260403_185856.md`)에서 worker visibility, generated preview UX, repeated soak hardening은 이미 1차 완료 상태다.
- 현재 남은 핵심 갭은 `stderr_tail` 일부 노출까지만 가능하고, 사용자가 앱/CLI 안에서 더 긴 stdout/stderr를 바로 확인하기 어렵다는 점이다.
- approval, multi-team, burn-in 후보도 유효하지만 현재 사용자 막힘을 가장 직접적으로 줄이는 것은 log observability 보강이다.

## 후보 상세 분석
### 1. background worker log / tail visibility + full viewer UX
- 현재 상태
  - `attach` / `status`는 `stdout_log=`, `stderr_log=`, `stderr_tail=`까지만 보여 준다.
  - project builder / TUI는 worker 상태와 일부 stderr preview는 보이지만, 더 긴 로그를 직접 탐색하는 표면은 없다.
- 가치
  - stuck, schema mismatch, prompt failure, runtime launch 문제를 사용자 표면에서 바로 진단할 수 있다.
  - 이미 저장되는 worker log metadata를 재사용하므로 새로운 저장 모델 변경이 거의 없다.
- 리스크
  - 큰 로그 파일을 그대로 읽으면 메모리/렌더링 비용이 커질 수 있으므로 bounded tail 정책이 필요하다.
- 판정
  - **가장 먼저 착수할 후보**

### 2. approval UX polish
- 현재 상태
  - approval modal과 approve/deny command round-trip은 이미 구현되어 있다.
  - 다만 요청 상세 맥락, decision preview, feedback 입력, 유형별 가독성은 아직 얇다.
- 가치
  - human-in-the-loop 경험을 개선한다.
- 리스크
  - permission/plan/sandbox 결정 흐름을 모두 건드릴 가능성이 있어 로그 UX보다 변경 반경이 넓다.
- 판정
  - **다음 우선순위 후보**

### 3. multi-team picker / overview 개선
- 현재 상태
  - TUI에 기본 team list는 있지만 이름과 member count 위주다.
  - team별 result 상태, active worker, approvals, last activity, workspace 요약은 한 화면에 모이지 않는다.
- 가치
  - 여러 팀을 동시에 운영할 때 효율이 커진다.
- 리스크
  - dashboard 집계와 팀 선택 UI를 함께 확장해야 해서 단일 팀 UX 개선보다 범위가 크다.
- 판정
  - **운영 scale-up 단계의 후보**

### 4. longer burn-in 결과 축적 / release checklist
- 현재 상태
  - soak harness, `latest-summary.json`, manual burn-in 문서까지는 있다.
  - 더 긴 누적 실행과 release checklist 체계화는 아직 남아 있다.
- 가치
  - 출시 신뢰도와 회귀 추적력이 올라간다.
- 리스크
  - 제품 체감 개선보다 운영 시간과 문서화 비용 비중이 크다.
- 판정
  - **UX 개선 이후 진행할 후보**
